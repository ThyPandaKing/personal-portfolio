## Overview
RL-Based HPC Scheduler with Runtime Estimation is a B.Tech final-year thesis project (IIT Tirupati, May 2023, by Aditya Sharma and Saurav Gwalia, supervised by Dr. Ajin George Joseph and Dr. Raghavendra Kanakagiri). It designs a Reinforcement Learning batch-job scheduler for High-Performance Computing (HPC) clusters and, crucially, replaces the unreliable user-supplied job runtime in the scheduler's state with a machine-learning-predicted runtime - improving scheduling quality without any human intervention.

## Problem / Motivation
Production HPC schedulers rely on hand-tuned heuristic priority functions. Once deployed, they cannot adapt to shifting job loads or changing optimization goals without error-prone expert re-tuning. RL-based schedulers (RLScheduler, DRAS) address adaptability by learning a policy that can be re-shaped simply by editing a reward function. However, their state space still depends on the user's estimated runtime for each job - a value users routinely over-state (often 30-40% too high) to avoid their jobs being killed. This inflated estimate corrupts the scheduler's view of the world and degrades wait time and utilization. The project's thesis: if the runtime in the state space is accurate, the same RL scheduler performs substantially better.

## How it works / Architecture
The system is a two-stage pipeline. Stage one is a runtime estimator: each Python job is tokenized with Python's lexical tokenizer, embedded into a fixed 25-element vector using NLP techniques (so structurally similar programs map to nearby vectors, following the user2code2vec idea), and hashed to a single integer code representation. That code value, together with the job's requested processors and memory, is fed to a Gradient Boosting Regression model that predicts the job's runtime. Trained over a 42k-program Python corpus, the estimator reaches ~90% accuracy (predictions within ~12% of the true runtime). Stage two is the scheduler: a Deep Q-Learning agent (built on the DRAS design) takes the queue of waiting jobs as state - now carrying the predicted runtime instead of the user estimate - and outputs Q-values per candidate job, selecting which to run, reserve, or backfill. Training runs on the CQSim HPC simulator with an epsilon-greedy schedule (epsilon = 1.0 decaying at 0.995) shifting from exploration to exploitation, and a reward function that balances average wait time, job size, and system utilization (weights 1/3 each) while discouraging starvation.

## Key features
A code-embedding runtime predictor that turns source code plus resource requests into an estimated runtime; a Gradient Boosting Regression model tuned by grid search (learning rate 0.1, max depth 8, 1000 estimators); a Deep Q-Learning scheduler with resource reservation and neural-network-driven backfilling; a tunable reward function so administrators can re-prioritize wait time vs. utilization vs. starvation at runtime; and an evaluation harness on both the real KTH-SP2-1996-2 trace and a synthetic trace derived from the Python corpus.

## Tech stack
Python for the full pipeline; Gradient Boosting Regression (machine-learning runtime estimation); NLP tokenization and code embeddings (Python tokenizer, hashing to an integer code id); Deep Q-Learning / Deep Neural Networks for the RL scheduling agent; and the CQSim HPC simulator for training and evaluation. Standard SWF job traces (KTH-SP2, SDSC-SP2, PIK-IPLEX) are used as workloads.

## Results
Against the DRAS baseline that uses user-estimated runtimes, feeding the ML-approximated runtime into the same RL scheduler improved average wait time by 42.35%, max runtime (a proxy for job starvation) by 53.66%, and system utilization by 9.2%. The RL scheduler also clearly outperformed a classical First-Come-First-Served baseline across all three metrics.

## Outcome
The work demonstrates that the accuracy of the runtime parameter in an RL scheduler's state space - not just the network architecture or training tricks - is a powerful and under-explored lever for HPC scheduling performance. As an academic capstone it validates the idea end-to-end in simulation; the authors note it can be strengthened further by tightening the link between the estimator and the live scheduler and by richer NLP code-embedding techniques.
